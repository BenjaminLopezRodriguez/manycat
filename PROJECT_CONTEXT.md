# manycat — PROJECT_CONTEXT.md

Generated by repo audit. Paste this into a new Claude session for full context.

## 1. What this is

manycat is a chat-UI wrapper around a sandboxed coding agent. The product surface is styled like WhatsApp (contacts list, message bubbles, unread badges) but each "chat" is a **workflow**: a user sends a prompt describing a code change, an agent works in an isolated sandbox container against a repo checkout, and the result is presented as a diff the user can **approve** or **request changes** on, inline in the chat thread and in a Monaco-based workspace panel. There is no WhatsApp Business API / Twilio integration anywhere in the codebase — "WhatsApp-style" describes the UI metaphor only.

**Core user loop:** open a workflow → type a request → agent runs in a sandbox → agent posts status updates + a diff message → user approves (marks workflow done) or requests changes (loops back) → repeat.

## 2. Architecture

- **Frontend/API**: Next.js 15 (App Router) + React 19, tRPC v11 (`@trpc/server`, `@trpc/react-query`), TanStack Query, Tailwind v4, shadcn/base-ui components, Monaco editor for the workspace/diff view.
- **DB**: Postgres via Drizzle ORM (`postgres` driver). Schema currently only defines the T3-scaffold `post` table (`src/server/db/schema.ts`) — **no tables exist yet for workflows, messages, or diffs.**
- **Auth**: **None implemented.** No Kinde/Clerk/NextAuth/Auth0 code anywhere despite README boilerplate referencing NextAuth. All tRPC procedures are `publicProcedure`.
- **Agent runtime**: separate Python service (`agent-harness/`), FastAPI + LangChain/LangGraph `create_react_agent` ReAct loop, "Claude Code–style" assembled system prompt (`prompts/assembler.py`), filesystem + shell tools scoped to a per-workflow workspace dir.
- **Sandbox orchestrator**: separate Node/Express service (`infra/sandbox-orchestrator/`) using `dockerode` to spin up one Docker container per workflow, bind-mounting a host workspace dir, allocating a host port (4000–4999) for a live preview URL.
- **Hosting/deploy**: Docker Compose for local dev (`docker-compose.yml`), or local Kubernetes via `kind` (`k8s/base` + `k8s/overlays/{local,prod}`). No Vercel-specific config despite Vercel plugin context in this session — this app is containerized, not deployed via `vercel deploy`.
- **Queues**: none. Agent runs are synchronous HTTP request/response (Next.js tRPC mutation → orchestrator → agent-harness `/run`, all blocking).
- **WhatsApp integration**: **does not exist.** No Twilio, no Meta Cloud API, no webhook signature verification anywhere in the repo.

## 4. Data model

**Persisted (Postgres via Drizzle, `src/server/db/schema.ts`):**
- `manycat_post` — `id` (identity PK), `name` (varchar 256), `createdAt`, `updatedAt`. T3-scaffold leftover; not used by the product. No relations.

**Not persisted — everything below lives only in client React state, seeded from static fixtures in `src/app/_fragments/chat/data.ts` and mutated in-memory by `agent-sim.ts`. There is no server-side storage of workflows, messages, or diffs.**

- `Workflow` — `id, name, initials, avatarClass, repo, status (idle|working|needs-review|done), unread?, messages: Msg[], workspace: WorkspaceFile[]`
- `Msg` (discriminated union) — `TextMsg`, `AgentStatusMsg`, `DiffMsg{path,before,after,summary}`, `ApprovalMsg{resolved: boolean|null}`, `MilestoneMsg`
- `WorkspaceFile` — `path, contents, language?, edited?`
- `Contact` — `id, name, initials, avatarClass, mutuals` (feed/invite UI only)

Relationship: one `Workflow` has many `Msg` and many `WorkspaceFile`; a `DiffMsg`'s `path` corresponds to a `WorkspaceFile.path`. All of this is client-only — a page refresh loses everything except the hardcoded seed in `data.ts`.

## 5. API surface

**tRPC router** (`src/server/api/root.ts`), mounted at `POST/GET /api/trpc/[trpc]`. All procedures are `publicProcedure` — no auth.

| Router.procedure | Type | Auth | Purpose |
|---|---|---|---|
| `post.hello` | query | none | T3 scaffold demo |
| `post.create` | mutation | none | T3 scaffold demo, writes to `manycat_post` |
| `post.getLatest` | query | none | T3 scaffold demo |
| `workflow.isEnabled` | query | none | Returns whether `AGENT_HARNESS_URL` + `SANDBOX_ORCHESTRATOR_URL` are set (`src/server/api/routers/workflow.ts:45`) — gates mock vs. real agent client-side |
| `workflow.createSandbox` | mutation | none | Proxies to orchestrator `POST /sandboxes` |
| `workflow.getSandbox` | query | none | Proxies to orchestrator `GET /sandboxes/:id` |
| `workflow.deleteSandbox` | mutation | none | Proxies to orchestrator `DELETE /sandboxes/:id` |
| `workflow.runAgent` | mutation | none | Creates sandbox, calls agent-harness `POST /run`, synthesizes a fixed event sequence (status → append → diff → approval) back to the client |

**Sandbox orchestrator** (`infra/sandbox-orchestrator/src/index.js`, Express, no auth):
| Method | Path | Purpose |
|---|---|---|
| GET | `/health` | liveness + sandbox count |
| POST | `/sandboxes` | create/reuse a Docker container for `workflowId`, bind-mount workspace, allocate port 4000–4999 |
| GET | `/sandboxes/:id` | inspect container status |
| DELETE | `/sandboxes/:id` | stop + remove container, release port |

**Agent harness** (`agent-harness/programming_agent/server.py`, FastAPI, no auth):
| Method | Path | Purpose |
|---|---|---|
| GET | `/health` | liveness |
| POST | `/run` | runs the LangGraph ReAct agent (`prompt`, `workflow_id`, `mode`) against the workspace, returns raw text `output` |

No REST routes under `src/app/api/` other than the tRPC catch-all. No webhooks, no cron routes.

## 6. Key flows

**(a) "Inbound message" → agent invocation**
There is no inbound channel (no WhatsApp/SMS/email webhook). The trigger is the in-app chat input (`chat.tsx:208` `send()`), which calls `agent.run(text)` from `useAgent()` (`agent-sim.ts:284`). `useAgent` picks `useRemoteAgent` if `workflow.isEnabled` is true, else `useMockAgent`. Remote path: `useRemoteAgent.run` → `api.workflow.runAgent.mutate` → `workflowRouter.runAgent` (`workflow.ts:98`) → orchestrator `POST /sandboxes` (ensures container) → agent-harness `POST /run` → `ProgrammingAgentHarness.run()` (`harness.py:85`) builds a LangGraph ReAct agent with filesystem/shell tools scoped to the workspace dir and invokes it once with the user's prompt.

**(b) Agent diff generation → storage → rendering**
The agent-harness returns only `{ output: string }` — a final text message, **not a structured diff**. `workflowRouter.runAgent` (`workflow.ts:174-198`) does not inspect the actual file changes the agent made; it unconditionally appends a hardcoded `ApprovalMsg` ("Review agent changes in the workspace — approve when ready.") and sets status to `needs-review`, regardless of whether the agent changed anything. **There is no real `DiffMsg` construction in the remote path** — the rich before/after diff UI (`DiffMsg`, `InlineDiffEditor` in `message-list.tsx`) is only ever populated by `useMockAgent`'s scripted fixtures in `data.ts` (`agentScripts`). Nothing is stored server-side; `handleAgentEvent` in `chat.tsx:115` mutates local React state only.

**(c) Approve / request-changes → state transitions → git side effects**
Both mock and remote `approve()`/`requestChanges()` (`agent-sim.ts:153`, `:175`, `:240`, `:258`) are pure client-side state mutations: `resolve-approval` event flips `ApprovalMsg.resolved`, appends a `MilestoneMsg` or follow-up text, and sets workflow `status` to `done` or `idle`. **No git operations occur.** The agent's shell tool (`filesystem.py`) is described as usable "for git, tests, builds" inside the sandbox, but nothing in the approve flow triggers a commit, push, or PR — git usage, if any, is entirely at the discretion of the LLM agent during its own tool loop, not a system-driven side effect of approval.

## 8. Unfinished / untested / risky

- **No persistence for the core product**: workflows/messages/diffs live only in client React state seeded from a static fixture. Refresh = data loss. Drizzle/Postgres is wired up but only backs the unused scaffold `post` table.
- **No auth anywhere**: every tRPC procedure is `publicProcedure`. Anyone hitting `/api/trpc` can trigger sandbox creation/deletion and agent runs for any `workflowId`.
- **Remote diff flow is a stub**: `workflowRouter.runAgent` (`src/server/api/routers/workflow.ts:174-198`) fabricates a generic approval message and never parses the agent's actual file edits into a `DiffMsg` — the polished diff UI only works with the mock agent.
- **Approve/request-changes have no backend effect**: no commit, no PR, no persisted approval record — purely local UI state, so approvals don't survive a refresh and there's no audit trail.
- **`workflow.deleteSandbox` / orchestrator `DELETE /sandboxes/:id`**: no corresponding automatic cleanup on approve/done or on error — a workflow that errors mid-run can leak a running sandbox container (`createSandbox` provisions but nothing in the failure path in `workflow.ts:157-171` calls delete).
- **Orchestrator has zero auth or input hardening beyond `safeId()`** path sanitization; any caller on the network can create/delete arbitrary Docker containers via `SANDBOX_IMAGE`.
- **`env.js` incomplete**: several env vars used by the orchestrator/agent-harness aren't validated by `@t3-oss/env-nextjs` (expected, since those are separate services/languages — but there's no equivalent schema validation in either service; agent-harness uses raw `os.getenv` with silent defaults like `MODEL=gpt-4o`).
- **No webhook signature verification code exists** — consistent with there being no webhooks/inbound integrations at all in this repo, contrary to any WhatsApp-specific expectation.
- **`agent.cancel()` in `useRemoteAgent`** (`agent-sim.ts:277`) is a no-op comment (`/* remote runs are not cancellable in v1 */`) — a long-running or stuck agent request can't be aborted from the UI.
- **Production k8s path documented but not hardened**: `docs/infra.md` explicitly flags that the orchestrator mounts the host Docker socket (dev-only pattern) and that production should use the Kubernetes Job API instead — not yet implemented (`k8s/base/sandbox-job-template.yaml` exists as a template only).
- **README is stock create-t3-app boilerplate**, referencing NextAuth/Prisma which aren't actually used — mildly misleading for onboarding.
